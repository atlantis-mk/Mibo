# Mibo 里程碑文档

本文档将 `Mibo` 定义为一个“类似 Emby 的家庭媒体系统”来规划开发里程碑。

目标不是一次性做完所有能力，而是按用户可感知的体验逐步交付：

- 能接入存储
- 能建立媒体库
- 能识别媒体与元数据
- 能播放
- 能记录多用户进度
- 能在性能压力出现时继续演进

本文档基于 `docs/media-architecture/improved-architecture.md` 的结论：

- 保留 `OpenList`
- 保留 `mibo-media-server` 作为业务核心
- 在 `mibo-media-server` 内建立 `StorageProvider` 抽象
- API 与 Worker 逻辑分离，但 V1 部署保持简单

## 产品目标

对标 Emby，需要先覆盖以下核心体验：

- 统一挂载本地磁盘、NAS、云存储
- 自动扫描电影、剧集等媒体库
- 自动抓取元数据、海报、季集信息
- 提供媒体详情页、播放地址、播放状态
- 记录观看进度、继续观看、已观看状态
- 面向 Web / Mobile / TV 提供统一 API

不作为初期目标的能力：

- 一开始就做完整转码平台
- 一开始就做 Live TV / DVR
- 一开始就做插件生态
- 一开始就做复杂推荐系统
- 在没有性能证据前绕过 `OpenList`

## 里程碑原则

- 每个里程碑都必须形成可验证的用户价值
- V1 先做“能用”，再做“像 Emby”
- 优先交付电影 / 剧集主路径，不先铺满所有媒体类型
- 没有明确收益前，不提前建设未来适配器或复杂基础设施

## Milestone 0: 架构基线

目标：让系统具备后续迭代的稳定骨架。

范围：

- 定义 `mibo-media-server` 的基础模块边界：
  - API
  - Worker
  - Library
  - Playback
  - Progress
  - Search
- 定义 `StorageProvider` 接口
- 实现 `OpenList Adapter`
- 建立基础数据模型：
  - libraries
  - media_sources
  - media_items
  - media_files
  - jobs
- 完成基础配置、日志、健康检查、错误模型
- 选定并接入 `SQLite` 或 `Postgres`

完成标准：

- 服务可以启动并完成健康检查
- `mibo-media-server` 可以通过 `OpenList Adapter` 访问存储
- 数据库中具备最小可用的媒体库与任务表结构

暂不包含：

- 元数据抓取
- 播放体验
- 用户系统
- 转码

## Milestone 1: 媒体库可用

目标：让用户可以真正“导入并看到自己的库”。

范围：

- 媒体库管理：
  - 创建媒体库
  - 绑定存储路径
  - 指定库类型，例如 Movies / TV Shows
- 扫描能力：
  - 全量扫描
  - 增量扫描
  - 手动重扫
- 文件分类：
  - 识别电影
  - 识别剧集
  - 建立文件与条目关联
- 基础浏览 API：
  - 库列表
  - 条目列表
  - 条目详情基础信息

完成标准：

- 用户可以添加一个库并扫描出可浏览的媒体条目
- 电影与剧集可以在数据库中稳定落库
- Worker 扫描任务失败时可追踪、可重试

暂不包含：

- 精准元数据匹配
- 精美详情页信息
- 观看进度

## Milestone 2: 元数据与详情体验

目标：让媒体库从“文件列表”变成“像 Emby 的媒体库”。

范围：

- 接入 `TMDB/TVDB` 等元数据源
- 元数据匹配流程：
  - 自动匹配
  - 低置信度待确认
  - 手动重匹配
- 丰富条目详情：
  - 标题
  - 原始标题
  - 简介
  - 海报 / 背景图
  - 年份
  - 类型
  - 演员 / 导演
  - 季 / 集结构
- 接入 `ffprobe`：
  - 视频编码
  - 音频轨道
  - 字幕轨道
  - 分辨率
  - 时长

完成标准：

- 大多数电影 / 剧集可自动匹配正确元数据
- 详情页 API 足以支撑 Emby 风格详情展示
- 文件技术信息可以写入并查询

暂不包含：

- 实时转码
- 复杂推荐

## Milestone 3: 播放闭环

目标：让用户真正能够从媒体库进入播放。

范围：

- 播放相关 API：
  - 获取播放源
  - 获取媒体文件链接
  - 获取媒体技术信息
- 初期播放策略：
  - 优先直连 / 直链
  - 由 `OpenList` 提供文件访问能力
  - 服务端控制播放权限与媒体选择
- 字幕与音轨选择基础能力
- 播放前校验：
  - 文件存在
  - 可访问
  - 媒体信息完整

完成标准：

- Web 或其他客户端可以调用 API 播放媒体
- 电影与剧集具备稳定的播放入口
- 常见播放失败可以追踪到具体原因

暂不包含：

- 完整转码流水线
- 复杂设备兼容策略

## Milestone 4: 多用户与进度系统

目标：补齐 Emby 核心使用体验中的“人”和“状态”。

范围：

- 用户与会话基础能力：
  - 用户账号
  - 登录
  - 基础权限
- 进度系统：
  - 播放进度同步
  - 已观看 / 未观看
  - 继续观看
  - 最近播放
- 用户维度数据：
  - 收藏
  - 喜欢
  - 播放历史
- API 访问隔离：
  - 按用户返回进度状态
  - 按用户返回历史与继续观看列表

完成标准：

- 多个用户使用同一媒体库时，状态互不污染
- 客户端首页可以展示“继续观看”和“最近播放”
- 登录后的用户体验开始接近 Emby 的基础形态

暂不包含：

- 家庭共享的复杂权限模型
- 管理后台的细粒度 RBAC

## Milestone 5: 搜索、首页与可用性增强

目标：让系统从“能用”进入“好用”。

范围：

- 搜索能力：
  - 标题搜索
  - 人物搜索
  - 剧集 / 电影过滤
- 首页聚合：
  - 最近新增
  - 继续观看
  - 最近播放
  - 按库浏览
- 后台作业体验：
  - 扫描状态可视化
  - 元数据修复任务
  - 失败任务重试
- 缓存优化：
  - 元数据缓存
  - 链接缓存

完成标准：

- 用户能快速找到内容而不是只靠目录浏览
- 首页具备基本媒体中心体验
- 运维可以观察扫描、匹配、探测任务状态

## Milestone 6: 性能与兼容性增强

目标：在真实使用压力下补齐 Emby 风格系统需要的工程能力。

范围：

- 将 API 与 Worker 做更清晰的资源隔离
- 在必要时引入 `Redis` 作为任务协调或缓存组件
- 对扫描热点场景做优化
- 对播放热点场景做优化
- 仅在有明确收益时评估新增：
  - `Direct Local Adapter`
  - `Direct SMB/NFS Adapter`
  - `Direct Cloud Adapter`
- 评估并补充基础转码能力：
  - 容器转换
  - 码率降级
  - 设备兼容兜底

完成标准：

- 大库扫描不会明显拖垮 API
- 播放热点与后台任务可以分离控制
- 是否绕过 `OpenList` 由性能数据驱动，而不是预设驱动

## Milestone 7: 高级 Emby 能力

目标：在核心链路稳定后，逐步补充高级能力。

候选范围：

- 更完整的转码能力
- 下载与离线支持
- 设备能力识别
- 多版本媒体选择
- 家庭共享权限增强
- 插件扩展点
- 直播电视 / DVR

说明：

- 这些能力不应阻塞前面里程碑
- 只有在基础体验稳定后才值得投入

## 推荐交付顺序

建议按以下顺序推进：

1. `M0` 架构基线
2. `M1` 媒体库可用
3. `M2` 元数据与详情体验
4. `M3` 播放闭环
5. `M4` 多用户与进度系统
6. `M5` 搜索、首页与可用性增强
7. `M6` 性能与兼容性增强
8. `M7` 高级 Emby 能力

## V1 定义

如果目标是尽快做出一个可用的 Emby-like V1，那么建议把 V1 定义为：

- `M0 + M1 + M2 + M3 + M4`

也就是：

- 可接入存储
- 可扫描媒体库
- 可展示元数据详情
- 可实际播放
- 可记录多用户进度

达到这一步时，系统已经具备“家庭媒体中心”的核心价值。

## 当前建议

如果现在开始执行，最优先的实现顺序应聚焦在：

1. `StorageProvider` + `OpenList Adapter`
2. `library / media_items / jobs / scanner`
3. 元数据匹配与 `ffprobe` 补全
4. 播放 API
5. 用户与进度系统

这样做可以最快把系统从“架构想法”推进到“真正可用的 Emby-like 产品”。
