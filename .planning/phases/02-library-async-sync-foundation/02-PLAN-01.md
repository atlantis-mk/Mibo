<objective>

Build the async library scanning foundation: global refresh interval settings, scheduled scan scheduling in the worker, library status badge on library cards, and a Jobs list view with retry capability.

</objective>

<context>

## Phase 2 Context

Administrators can connect storage-backed libraries and trust scans/refreshes to run asynchronously without degrading interactive requests. Admins can add media sources (local/NAS/cloud), create libraries bound to sources, trigger scans that queue and process in background, and configure global scheduled refreshes.

## Key Decisions from Context

- **D-01**: Hybrid approach — library status badge on library cards for quick feedback, plus Jobs list view for detailed monitoring and retry
- **D-02**: Global refresh interval — one system-wide interval applies to all libraries, not per-library schedules
- **D-03**: Merge behavior on rescans — add new, update changed, soft-delete missing. Full rebuild available as explicit admin action only
- **D-04**: Local storage sources need only provider + root path. No additional V1 config

## Existing Code

- `mibo-media-server/internal/worker/worker.go`: Worker polls every 2s, handles `sync_library`, `match_media_item`, `probe_media_file`
- `mibo-media-server/internal/jobs/service.go`: Job queue with Enqueue, EnqueueUnique, Claim, Complete, Fail, Retry
- `mibo-media-server/internal/library/service.go`: CreateLibrary already queues scan job on creation
- `mibo-media-server/internal/library/scan.go`: Scan logic with merge behavior, fingerprint-based change detection
- `mibo-media-server/internal/httpapi/router.go`: `POST /api/v1/libraries/{id}/scan`, `GET /api/v1/jobs`, `POST /api/v1/jobs/{id}/retry` already exist

## Research Findings

1. Job queue system is fully implemented with deduplication via `job_key`
2. Library scan status transitions: `pending → syncing → active | error`
3. Missing: settings service for global refresh interval, scheduled scan trigger in worker
4. Missing: frontend library status badge, Jobs list with filtering and retry
5. HTTP endpoints already exist for scan trigger, job listing, job retry

</context>

## Task 2: Backend — Scheduled Scan Scheduling in Worker

**Type:** execute
**Files:** `mibo-media-server/internal/worker/worker.go`
**read_first:**
- `mibo-media-server/internal/worker/worker.go`
- `mibo-media-server/internal/settings/service.go` (from Task 1)
- `mibo-media-server/internal/library/service.go` (for QueueLibraryScan)

**Action:**
1. Modify `NewRunner` to accept `*settings.Service` as a new dependency

2. Add scheduled scan ticker to `Run(ctx)`:
   ```go
   // Schedule scan ticker based on refresh interval
   refreshInterval := time.Duration(s.cfg.Worker.RefreshIntervalHours) * time.Hour
   if refreshInterval <= 0 {
       refreshInterval = 24 * time.Hour // default
   }
   scanTicker := time.NewTicker(refreshInterval)
   defer scanTicker.Stop()
   ```

3. On each `scanTicker.C` tick:
   - Query all libraries with `status = 'active'` and `scanner_enabled = true`
   - For each library, call `library.QueueLibraryScan(ctx, library.ID)` (uses EnqueueUnique so manual scans deduplicate)
   - Log: `"scheduled scan triggered for library {id}"`

4. Add `RefreshIntervalHours int` to `config.WorkerConfig`

5. On startup, read `RefreshIntervalHours` from settings service to initialize the ticker

**acceptance_criteria:**
- Worker logs "scheduled scan triggered for library {id}" when ticker fires
- Changing `refresh_interval_hours` via API and restarting worker picks up new interval
- Manual scans use EnqueueUnique so they don't duplicate scheduled scans for same library
- If refresh_interval_hours is 0 or missing, scheduled scanning is disabled (worker continues without scanTicker)
