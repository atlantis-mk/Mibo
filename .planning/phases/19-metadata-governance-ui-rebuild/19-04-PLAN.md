<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| image and asset actions -> mutation endpoints | user selection and repair clicks mutate governance state |
| asset status / playability data -> operator decisions | backend-derived availability/probe state influences repair actions |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-19-11 | T | image selection | mitigate | restrict selection actions to candidates already returned by the normalized hook and mutate only the selected flag for one item/image-type group |
| T-19-12 | E | asset link repair | mitigate | require link/unlink actions to target one explicit `itemId + assetId + role + segment_index` combination so the UI cannot widen repairs accidentally |
| T-19-13 | I | asset explainability panel | mitigate | show only contract-safe asset metadata, status, probe state, and reason codes; do not render raw storage-provider payloads or filesystem internals |
| T-19-14 | R | destructive unlink action | mitigate | gate unlink through an explicit confirmation step and keep mutation feedback visible in the detail shell so operators can trace what changed |
</threat_model>
