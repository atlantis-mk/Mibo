## Context

OpenList's file response model includes `thumb` on both list and get responses. In upstream OpenList, `server/handles/fsread.go` defines `ObjResp.Thumb string json:"thumb"`; `FsGetResp` embeds `ObjResp`, so `/api/fs/get` can return `thumb` alongside `raw_url`, `provider`, `related`, and hash metadata. The value comes from `model.GetThumb(obj)`, which depends on the underlying OpenList driver. Several cloud drivers implement thumbnail behavior, but the field is not guaranteed for every provider and it does not distinguish poster from backdrop.

Mibo's current OpenList adapter parses `name`, `is_dir`, `size`, `modified`, `raw_url`, `provider`, and `hash_info`, but not `thumb`. Mibo's artwork fallback path already protects non-generated remote artwork, then uses sibling image files, then ffmpeg frame extraction. This change should insert OpenList `thumb` into that fallback ladder without treating it as authoritative metadata.

## Goals / Non-Goals

**Goals:**

- Preserve the current highest priority for TMDB and other non-generated selected artwork.
- Use explicit sibling artwork files before generic OpenList thumbnails.
- Use OpenList `thumb` before ffmpeg frame extraction when no stronger item-appropriate image source exists.
- Avoid using generic thumbnails as backdrops unless a future explicit setting or provider signal supports that safely.
- Keep storage metadata provider-neutral so future adapters can expose thumbnail-like fields without OpenList-specific branching in catalog code.
- Keep all behavior backend-only and transparent to existing frontend consumers.

**Non-Goals:**

- This change does not modify upstream OpenList.
- This change does not make OpenList generate thumbnails if the underlying driver does not provide them.
- This change does not classify OpenList `thumb` as a reliable backdrop/background image.
- This change does not add a frontend setting for custom artwork priority.
- This change does not change TMDB matching, catalog governance, or frontend presentation contracts.

## Artwork Priority Order

The recommended artwork source ranking is:

1. Existing non-generated selected artwork, including TMDB poster/backdrop and manually selected remote artwork.
2. Same-directory explicit artwork files found through the active storage provider.
3. OpenList/provider thumbnail metadata for item-appropriate primary artwork: movie `poster`, episode `still`.
4. ffmpeg-generated fallback poster and backdrop from video frames.

Detailed behavior:

- Movie poster selection should use `thumb` only if the movie has no non-generated poster and no sibling poster/cover/folder image.
- Episode still selection should use `thumb` only if the episode has no non-generated still image. This matches frontend episode presentation, which prefers `still`/`backdrop` rather than an episode `poster`.
- Backdrop selection should not use `thumb` by default. It should continue to prefer sibling `backdrop`, `background`, or `fanart` images, then generated frame extraction.
- If a sibling poster exists but no sibling backdrop exists, poster should come from the sibling image while backdrop may still be generated by ffmpeg.
- If OpenList `thumb` exists but a TMDB/non-generated poster or episode still already exists, `thumb` must be ignored for that image type.
- If OpenList `thumb` exists and only generated movie poster exists, `thumb` may replace the generated poster as a better provider-supplied fallback. For episodes, `thumb` should be stored as `still` so existing episode detail and season-list UI can display it.

This ordering keeps semantic accuracy ahead of convenience: explicit metadata and explicit files beat generic thumbnails; generic thumbnails beat CPU-heavy/generated frame extraction.

## Decisions

### Add provider-neutral thumbnail metadata

Extend `storage.Object` with an optional field such as:

```go
ThumbnailURL string `json:"thumbnail_url,omitempty"`
```

The OpenList adapter should parse `thumb` from both `/api/fs/list` and `/api/fs/get` into this field. Catalog/probe code should read `storage.Object.ThumbnailURL`, not OpenList-specific response structs. This keeps the storage abstraction clean and allows future providers to expose thumbnails through the same field.

Alternative considered: store OpenList `thumb` inside `HashInfo` or provider-specific metadata. That avoids changing the object contract but hides a first-class presentation URL in an unrelated hash field and forces fragile key conventions.

### Use `thumb` as primary artwork, not backdrop

OpenList `thumb` is a generic thumbnail. Its aspect ratio, resolution, crop behavior, and source vary by driver. The first implementation should map it to `poster` for movies and `still` for episodes, because episode presentation already reads `still` before falling back to `backdrop`. Backdrops should remain explicit sibling images or ffmpeg-generated frames.

Alternative considered: use `thumb` for both poster and backdrop. This would reduce generated background work but risks stretching/cropping low-resolution or portrait thumbnails as hero backgrounds.

### Keep fallback artwork ownership in `probe/artwork.go`

The artwork fallback source ordering already lives in `internal/probe/artwork.go`. Adding thumbnail selection there keeps the decision close to sibling artwork and frame extraction and avoids widening scanner responsibilities. The scan path still queues video files; artwork enrichment remains asynchronous with probe work.

Alternative considered: set item artwork during library scanning from OpenList list responses. That would make first scan faster for thumbnails present in directory listings, but it would duplicate artwork selection logic and complicate catalog item/asset linkage before probe resolution.

### Persist provider thumbnails as selected image URLs

When selected, the OpenList thumbnail URL should be stored in `database.ItemImage.URL` with `ImageType = "poster"` for movies or `ImageType = "still"` for episodes, and `IsSelected = true`. If the URL is relative or signed, implementation should preserve the exact usable URL returned by OpenList. Existing HTTP response normalization should only transform Mibo-relative generated artwork URLs, not remote OpenList thumbnail URLs.

Alternative considered: proxy/copy OpenList thumbnails into `MIBO_ARTWORK_ROOT_PATH`. That gives stable serving behavior but adds download/cache invalidation complexity. Direct URL storage matches the existing remote artwork model.

## Implementation Outline

1. Extend `storage.Object` with `ThumbnailURL`.
2. Parse OpenList `thumb` from list/get responses in `internal/storage/openlist/adapter.go`.
3. Add thumbnail selection in `internal/probe/artwork.go` after sibling artwork lookup and before ffmpeg extraction, choosing movie `poster` or episode `still` by catalog item type.
4. Reuse the existing selected-image upsert path so generated poster rows can be replaced but non-generated TMDB/manual rows are preserved by the existing guard.
5. Keep backdrop generation unchanged except for respecting existing sibling backdrop behavior.
6. Add tests for storage parsing, movie poster priority, episode still priority, generated-art replacement, remote-art preservation, and fallback to ffmpeg when `thumb` is empty.

## Risks / Trade-offs

- OpenList `thumb` availability varies by driver, so coverage improvement is opportunistic rather than guaranteed.
- Thumbnail URLs may be signed or short-lived depending on provider. If this becomes a problem, a later change can add Mibo-side caching/proxying.
- Generic thumbnails may not look like poster art. Ranking them below explicit sibling images and TMDB minimizes quality regressions.
- Some providers may expose image thumbnails for videos only after OpenList cache refresh. Mibo should not fail probe/artwork generation if `thumb` is absent.
- Direct remote thumbnail URLs may require auth in some deployments. Existing raw/link URL handling has similar provider constraints; tests should cover blank/malformed URLs gracefully.

## Migration Plan

No database migration is required. Existing `item_images` rows remain valid. New scans or reprobe jobs can select OpenList thumbnails for items that currently have only generated poster artwork or no poster artwork.

Rollback is code-level: removing thumbnail parsing and selection will leave already stored thumbnail URLs in `item_images`. They can remain harmless selected remote artwork or be cleaned by a future maintenance action if needed.

## Open Questions

- Should Mibo eventually cache OpenList thumbnails to avoid signed URL expiry?
- Should a future admin setting allow using provider thumbnails as backdrops for specific trusted drivers?
- Should thumbnail source metadata be recorded separately for governance/debugging, or is the stored URL sufficient for the first implementation?
