## Context

当前系统已经具备媒体库、catalog 投影、播放资源映射和基础角色模型，但角色主要用于管理操作授权，内容读取链路仍默认对已登录用户开放。媒体内容的展示与播放广泛依赖 `library_id`、`resource_library_links` 和 `library_metadata_projections`，因此按库范围做访问控制比按元数据条目或单个视频资源打权限标签更契合现有结构。

这次变更是一个跨模块改动：它同时影响媒体库配置、角色定义、认证后的授权解析、catalog 读路径和播放入口。除了后端数据模型变化，还需要前端提供库标签和角色规则的管理能力，并在 UI 上明确提示“未打标签库默认开放”。

## Goals / Non-Goals

**Goals:**
- 为媒体库建立可管理的标签模型，支持将多个标签绑定到单个库。
- 为角色建立 allow/deny 两类库标签规则，并在用户拥有多个角色时稳定聚合。
- 在请求进入后统一解析用户可访问的库集合，并复用于 catalog、详情资源和播放授权路径。
- 明确默认访问策略，保证功能按“默认开放、按规则收紧”的方式渐进启用。
- 在同一元数据条目跨多个库存在时，保持展示体验可用，同时避免资源和播放穿透到不可访问库。

**Non-Goals:**
- 不引入用户级自定义覆盖规则，访问控制仅由角色驱动。
- 不实现标签继承、标签表达式或层级规则引擎。
- 不将影片、剧集或资源本身的元数据标签直接用于访问控制。
- 不在首版中引入预计算的用户可访问内容投影表。

## Decisions

- 使用“库标签”而不是“内容标签”作为访问控制输入。
  - Rationale: 现有 catalog 和播放链路普遍已经按 `library_id` 聚合与过滤，基于库做授权可以最小化对扫描、识别和元数据治理链路的扰动。
  - Alternative: 直接给 metadata item 或资源打访问标签；被拒绝，因为会让权限逻辑渗入内容语义层，并增加跨库重复内容的处理复杂度。

- 角色规则采用 `allow` / `deny` 双通道模型，并规定 `deny` 优先。
  - Rationale: 这同时支持“只能看某些库”和“不能看某些库”两类业务场景，且冲突处理直观。
  - Alternative: 只支持 allow 或只支持 deny；被拒绝，因为难以表达例外场景，后续很可能返工。

- 默认策略定义为“无标签库开放、无 allow 规则开放”。
  - Rationale: 该策略与现有系统行为最接近，避免在引入新能力时因未配置标签或角色规则造成内容意外消失。
  - Alternative: 默认拒绝；被拒绝，因为 rollout 风险高，配置遗漏会直接破坏现有可用性。

- 每个请求先解析“可访问库集合”，后续读取路径统一消费这一中间结果。
  - Rationale: 将规则求值与业务查询解耦，可以避免首页、浏览、收藏、详情和播放各自实现一套标签判断。
  - Alternative: 在每个查询里直接拼接角色标签规则；被拒绝，因为重复逻辑多，后续难以保持行为一致。

- 同一元数据条目在多个库中存在时，采用“展示按至少一个可访问库判断，资源与播放按可访问库严格裁切”的策略。
  - Rationale: 这既保留了跨库去重后的正常浏览体验，也避免详情资源和播放链路暴露不可访问库的数据。
  - Alternative: 只要任一来源库不可访问就整体隐藏；被拒绝，因为会让跨库重复内容的显示过于保守，损伤体验。

- 首版采用动态求值，不引入用户或角色维度的预计算内容可见性投影。
  - Rationale: 规则和库标签变更应立即生效，而当前规模下通过可访问库集合约束查询已足够清晰。
  - Alternative: 预计算“角色/用户可见内容表”；被拒绝，因为刷新链路复杂，且会放大跨模块一致性成本。

## Risks / Trade-offs

- [无标签库默认开放带来配置遗漏风险] → Mitigation: 在库管理界面明确标识“未设置访问标签，默认对所有角色可见”，并在后端保留这一默认策略说明。
- [读路径覆盖不完整导致权限穿透] → Mitigation: 将首页、浏览、收藏、继续观看、最近播放、详情资源和播放入口列为首版必须覆盖范围，并在测试中按入口逐项校验。
- [多角色与跨库重复内容的行为不易直觉理解] → Mitigation: 固化规则求值顺序和跨库策略，在设计与 UI 文案中明确 `deny` 优先和“资源仅来自可访问库”。
- [现有单字符串 `role` 与多角色模型并存] → Mitigation: 继续兼容现有主角色字段，但所有库可见性求值统一基于 `Roles` 集合执行。
- [查询路径增加库范围过滤后可能影响部分列表性能] → Mitigation: 首版以动态过滤上线，后续若出现明显瓶颈，再评估基于库范围的索引优化或预计算方案。

## Migration Plan

1. 新增库标签与角色规则的数据模型，并保持默认空配置对现有用户行为无影响。
2. 为角色管理和库管理 API 增加标签与规则读写能力，同时继续兼容现有角色管理功能。
3. 在认证成功后的后端请求链路中加入“可访问库集合”解析。
4. 逐步将 catalog、详情资源和播放入口切换为基于可访问库集合裁切。
5. 前端上线库标签与角色规则配置界面，并对无标签库显示默认开放提示。
6. 回滚时可先停用库可见性裁切逻辑，同时保留库标签和角色规则数据，避免数据损失。

## Open Questions

- 是否需要限制哪些角色可以配置库标签规则，还是保持现有 admin-only 管理入口即可？
- 库标签是否复用现有 `tags` 表并以 `kind=library` 区分，还是单独建立库访问标签模型以换取更清晰的语义边界？
- 当用户直接指定某个 `library_id` 请求详情或浏览时，系统返回“未找到”还是“无权访问”更符合现有 API 风格？
