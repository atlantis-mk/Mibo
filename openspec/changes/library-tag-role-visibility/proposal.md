## Why

当前系统的角色能力主要用于管理操作授权，还不能按内容来源控制用户可见范围。随着媒体库数量和内容分组需求增加，需要一种基于库标签的访问控制模型，让管理员能够按角色决定用户可以看到哪些库中的内容，并禁止访问特定标签对应的库。

## What Changes

- 为媒体库增加标签能力，支持复用已有库标签或创建新库标签，并将多个标签绑定到单个库。
- 为角色增加库标签可见性规则，支持配置允许访问的标签和禁止访问的标签。
- 基于用户角色聚合后的规则，计算用户可访问的库范围，并将其应用到内容读取链路。
- 将首页、浏览/搜索、收藏、继续观看、最近播放、详情资源和播放入口统一约束到可访问库集合。
- 明确默认访问策略：无标签库对所有人可访问；当角色未配置 allow 规则时，所有未命中 deny 规则的库均可访问；deny 优先于 allow。

## Capabilities

### New Capabilities
- `library-tag-role-visibility`: 为媒体库打标签，并基于角色的 allow/deny 规则控制用户可访问的库与内容范围

### Modified Capabilities
- `user-role-management`: 角色定义从管理操作授权扩展为可承载库标签可见性规则

## Impact

- 后端需要新增库标签、角色标签规则及用户可访问库范围解析能力。
- catalog、playback、library 相关 HTTP API 与服务层读取路径需要统一接入库可见性裁切。
- 前端需要在库管理和角色管理界面增加标签与可见性规则配置入口，并适配按权限裁切后的内容展示。
- 现有角色模型和登录态返回需要稳定提供用户角色集合，供后端统一计算访问范围。
